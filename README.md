# Контроль уборки банкоматов

Веб-приложение для планирования, исполнения и контроля уборки банкоматов.

**Текущая версия:** v2.4.7 — домен control-app.ru, SSL (Let's Encrypt), nginx + certbot.

Полная история: [CHANGELOG.md](./CHANGELOG.md)

## Возможности

- **Роли:** бизнес-администратор, администратор, супервайзер, исполнитель
- **Конструктор рабочего пространства** — для всех ролей: навигация, стартовая страница, виджеты дашборда (`/workspace`)
- **Справочники устройств** (бизнес-администратор) — Территориальный Банк, ГОСБ, Вид доступности
- Заявки на уборку с фотоотчётом (слева, справа, спереди)
- **Завершение заявки** — из списка, модалки «Завершить» или «Открыть»; исполнитель и **менеджеры** — со статусов «В работе», «Просрочено», «Возврат», «Экстренная»; **обязательны 3 фото**; CV — по ролям в настройках (bizadmin всегда без CV)
- **Данные при закрытии** — устройство, ОС, геолокация (запрос при входе, отображение в карточке заявки)
- **Фильтры заявок** — на мобильных сворачиваются по умолчанию; на десктопе — на всю ширину; фильтр по **ID УС**; у **исполнителя** — **8 вкладок по статусам** в нижней навигации (4×2) с цветами, иконками и счётчиками
- **Массовое назначение** — менеджеры при активных фильтрах выбирают заявки и назначают исполнителя; **исполнитель** — массово берёт новые заявки на себя
- **Сжатие фото в браузере** — до 1280px JPEG перед отправкой (снижает нагрузку на сервер)
- **Сжатие на сервере** (опционально, sharp) — для крупных файлов; на VPS рекомендуется `PHOTO_SKIP_SHARP=true`
- **CV-проверка фото** (вкл/выкл) — CLIP определяет банкомат Сбербанка; проверка **синхронно при загрузке**, очередь на сервере, предзагрузка модели при старте
- **Таймаут 120 с** для загрузки фото и завершения заявки (первая загрузка CLIP ~16–90 с)
- **Настройки CV** (бизнес-администратор) — вкл/выкл модели, порог, запас; **роли для CV** при завершении (admin, supervisor, executor); камера или галерея на mobile у исполнителя
- **Офлайн-режим (PWA)** — кэш интерфейса, сохранённые заявки, очередь фото и действий при отсутствии сети
- **Устойчивая загрузка** — таймаут API 20 с, кэш IndexedDB не блокирует экран; при сбое сети показывается ошибка или сохранённые заявки
- Push-уведомления и PWA — **требуется HTTPS** (кроме localhost); на iPhone/iPad push работает только из приложения на главном экране (iOS 16.4+)
- Integration API для ERP/CRM/1С

### Роли

| Роль | Код | Основные права |
|------|-----|----------------|
| Бизнес-администратор | `bizadmin` | Все права админа и супервайзера + раздел «Настройки» (CV) |
| Администратор | `admin` | Заявки, банкоматы, все пользователи (кроме bizadmin), интеграция |
| Супервайзер | `supervisor` | Заявки, устройства, управление исполнителями |
| Исполнитель | `executor` / `cleaner` | Свои заявки, фотоотчёт, «Взять заявку» (новые и **просроченные** нераспределённые); CV по роли |

## Деплой на VPS (Reg.ru и др.)

Приложение — **Node.js 22+** + Express. На production один процесс отдаёт API и собранный React.

```bash
git clone https://github.com/DjSurfTSI/control-app.git
cd control-app
sudo bash deploy/setup-server.sh
```

Скрипт установит зависимости, соберёт фронтенд, запустит **pm2** (`control-app` на порту `3001`) и настроит **nginx** как reverse proxy.

### Домен control-app.ru и SSL

**1. DNS в панели Reg.ru** (домен привязан к VPS):

| Запись | Тип | Значение |
|--------|-----|----------|
| `@` или `control-app.ru` | A | IP вашего VPS |
| `www` | A | тот же IP (или CNAME → `control-app.ru`) |

Подождите 5–30 минут после сохранения DNS.

**2. На сервере** (после `git pull`):

```bash
cd ~/control-app
sudo CERTBOT_EMAIL=ваш@email.ru bash deploy/setup-ssl.sh
```

Скрипт установит **certbot**, обновит nginx для `control-app.ru` / `www`, выпустит **Let's Encrypt** и включит редирект HTTP → HTTPS. Сертификат обновляется автоматически.

**3. Проверка:**

```bash
curl -I https://control-app.ru
pm2 logs control-app --lines 10
```

Откройте https://control-app.ru в браузере. Для Push-уведомлений нужен именно HTTPS.

Если порты закрыты файрволом: откройте **80** и **443** (Reg.ru → VPS → Сеть / firewall, или `sudo ufw allow 80,443/tcp`).

Переменные окружения — `server/.env` (шаблон: `server/.env.example`):

```env
PORT=3001
JWT_SECRET=длинный-случайный-секрет
VAPID_PUBLIC=...
VAPID_PRIVATE=...
CV_ENABLED=true
CV_ATM_THRESHOLD=0.30
CV_ATM_MARGIN=0.12
PHOTO_MAX_EDGE=1280
PHOTO_JPEG_QUALITY=82
PHOTO_SKIP_SHARP=true
PHOTO_PASSTHROUGH_MAX_BYTES=1800000
```

> **CV:** модель CLIP (~150 MB) **предзагружается** при старте сервера (`warmupCvModel`). Рекомендуется **≥1 GB RAM** или swap + `PHOTO_SKIP_SHARP=true`.
>
> **Настройки CV** (`enabled`, `threshold`, `margin`) — в UI (bizadmin) и БД `cv_settings`. Статус для UI: `GET /api/settings/cv/status`.

Обновление на сервере:

```bash
cd ~/control-app
git pull
npm install --prefix server   # при изменении server/package.json
bash deploy/build-client.sh   # обязательно после изменений client/
pm2 restart control-app
```

**Чеклист после `git pull`:**

1. `bash deploy/build-client.sh` — без `client/dist/` nginx отдаст старый или пустой UI.
2. `pm2 logs control-app --lines 15` — строка `Сервер запущен: http://localhost:3001`.
3. В браузере жёсткое обновление (Ctrl+Shift+R) или очистка кэша PWA.

### Ошибка сборки: `Could not resolve "./geolocation.js"`

```
error during build:
Could not resolve "./geolocation.js" from "src/utils.js"
```

**Причина:** неверный относительный путь к модулю геолокации.

| Файл | Путь модуля | Импорт |
|------|-------------|--------|
| `client/src/utils.js` | `client/src/utils/geolocation.js` | `'./utils/geolocation.js'` |
| `client/src/pages/Login.jsx` | тот же | `'../utils/geolocation'` |
| `client/src/context/AuthContext.jsx` | тот же | `'../utils/geolocation'` |

Исправление уже в `main` (v2.1.1). На сервере: `git pull` и повторить `bash deploy/build-client.sh`.

Локальная проверка перед деплоем:

```bash
npm run build --prefix client
```

### Офлайн-режим

Исполнитель может работать без интернета (после **первого входа** с сетью):

- Просмотр ранее загруженных заявок
- «Начать» / «Завершить» заявку — действия попадают в очередь
- Прикрепление фото — сохраняются на устройстве (📡), отправляются при появлении сети
- Баннер вверху показывает статус сети и кнопку «Синхронизировать» (с сообщением об успехе или ошибке)

Установите приложение на экран телефона (PWA) через меню браузера «Добавить на главный экран». На iPhone контент не заходит под часы и индикаторы сети (`safe-area-inset`, `viewport-fit=cover`).

### Push-уведомления

- Кнопка **🔕/🔔** в шапке (десктоп) или в нижней навигации (мобильные).
- **HTTPS обязателен** — на `http://IP` браузер блокирует Service Worker и Push.
- **iPhone/iPad:** добавьте на главный экран и откройте оттуда (не из вкладки Safari/Chrome); iOS/iPadOS 16.4+.
- На сервере: `sudo CERTBOT_EMAIL=admin@control-app.ru bash deploy/setup-ssl.sh` (см. `deploy/setup-ssl.sh`)

### Геолокация

- При **первом входе** (клик «Войти») браузер запрашивает доступ к геолокации — нужен для фиксации места закрытия заявки.
- Запрос запускается **сразу** при нажатии «Войти» (`requestGeolocationAccess()`), до завершения HTTP-логина — иначе браузер не покажет диалог (истекает user gesture).
- Координаты кэшируются в `localStorage` (`geo_position_cache`); при завершении заявки обновляются без повторного диалога (`refreshGeolocationIfGranted`).
- В карточке выполненной заявки — блок **«Данные при закрытии»** (устройство, ОС, ссылка на карту).
- **Требуется HTTPS** (или `localhost`); на HTTP диалог не появится.
- Модуль: `client/src/utils/geolocation.js`. Подробнее: [ARCHITECTURE.md](./ARCHITECTURE.md) → §7.1.

### Ошибка 502 (приложение не открывается)

Частые причины:

1. **Сервер Node.js не запустился** — синтаксическая ошибка в коде, сбой при старте. Проверка: `pm2 logs control-app --lines 20` (ожидается «Сервер запущен: http://localhost:3001»).
2. **pm2 не запущен** — `pm2 status control-app`, перезапуск: `pm2 restart control-app`.

```bash
cd ~/control-app && git pull && pm2 restart control-app
```

### Сборка падает с «Killed»

Сообщение `Killed` при `vite build` — **нехватка RAM** на VPS (OOM killer). Уязвимости `npm audit` в `concurrently` на это не влияют (это dev-зависимость корня репозитория, не production).

**Вариант 1 — включить swap на сервере (рекомендуется):**

```bash
sudo bash deploy/ensure-swap.sh
bash deploy/build-client.sh
pm2 restart control-app
```

**Вариант 2 — собрать на своём ПК и залить только `dist`:**

```bash
# на локальной машине
npm run build --prefix client
scp -r client/dist root@IP_СЕРВЕРА:~/control-app/client/
# на сервере
pm2 restart control-app
```

Проверка памяти: `free -h`

### Ошибка 502 при загрузке фото

Причина — нехватка RAM на VPS. С **v1.2.0** фото сжимаются в браузере; на сервере sharp часто не вызывается (`optimizePhoto passthrough`).

**Обязательно на сервере:**

```bash
cd ~/control-app
git pull
npm install --prefix server
bash deploy/build-client.sh

# В server/.env добавьте (или раскомментируйте):
# PHOTO_SKIP_SHARP=true

pm2 delete control-app
pm2 start deploy/ecosystem.config.cjs
pm2 save
sudo bash deploy/ensure-swap.sh
```

Логи при загрузке: `pm2 logs control-app --lines 30` — должна быть строка `optimizePhoto passthrough`.

Дополнительно: отключите CV в «Настройки» (bizadmin), если RAM < 1 ГБ.

### Бесконечная «Загрузка...» на экране заявок

С **v1.3.1** запросы к API ограничены по времени (20 с). Если сервер не отвечает, вместо вечной загрузки появится сообщение об ошибке; при наличии кэша показываются ранее загруженные заявки.

**Если проблема остаётся после обновления:**

```bash
cd ~/control-app
git pull
bash deploy/build-client.sh
pm2 restart control-app
```

В браузере: жёсткое обновление (Ctrl+Shift+R). Проверьте в DevTools → Network, отвечают ли `/api/auth/me` и `/api/tasks`. Логи сервера: `pm2 logs control-app --lines 50`.

### «Загрузка фото...» при открытии заявки

С **v1.3.2** слоты для прикрепления фото отображаются сразу — ожидание CV-настроек и списка фото больше не блокирует завершение заявки. При ошибке сети показывается кнопка «Повторить».
